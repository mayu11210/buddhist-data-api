## 現代語訳作業フロー（最優先・1 サイクル）

**1 篇単位で push まで完結させる。** idx=0 真済序・idx=1 山に遊んで仙を慕う詩
で確立済のフロー：

1. Cowork：JSON `data/kukai/shoryoshu_miyasaka.json[i].ページ[0].kakikudashi`
   から書き下しを読み出す
2. Cowork：下訳生成（不明箇所は [要確認] マーク・典故補注は括弧書きで濃密）
3. ケンシン：弘大六 PDF Gemini OCR 既訳をチャットに貼付
4. Cowork：段落単位で既訳と突合し差分整理
5. Cowork：「意味が通じる」基準で Cowork 訳を尊重・伝記矛盾／逐字誤読のみ
   既訳に寄せて修正
6. Cowork：JSON `ページ[0].gendaigoyaku` に Python 経由で反映
   （NULL バイト検証必須）
7. Cowork：commit_message.txt 更新（Python 経由）→「commit_push.bat ダブル
   クリックお願いします」と依頼
8. ケンシン：commit_push.bat 実行 → commit + push
9. push 成功確認 → 次篇へ

### 取込順序の原則
**idx 番号順に進める**（idx=0 → 1 → 2 → 3 ...）。「散文系推奨」のような
ラベル誘導で順序を飛ばさない。前メモの推奨候補も「同じ idx 範囲内での
選び方」程度に留め、巻番号や idx を飛び越えない。

### 現代語訳の品質方針

- **共有知識バンク方針**：補注は豊かに（典故・歴史背景・字義解釈）
- **直訳寄り**：並列構文を保つ・対句を崩さない
- **詩篇は各句 1 行ずつ訳出**（散文化しない）
- **典故補注は括弧書き**で濃密に（郭璞・葉公・北邙・王子晋・伯夷・老聃・
  許由・大鵬・崑崙蓬莱・遮那・三密・阿字 等）
- **密教教学用語は原語維持**（三密・心王・遮那・阿字・無縁慈悲 等）。既訳が
  通俗化していても Cowork 訳を維持
- **伝記矛盾／逐字誤読は必ず修正**（既訳の方が正しいケースを特定して採用）
- **韻律は現代語で再現困難ゆえ意味重視**

### 字数倍率の参考値（既存実績）

| 篇 | 詩形 | 書き下し | 現代語訳 | 倍率 |

| idx=1 山に遊んで仙を慕う詩 | 五言詩 | 1482 字 | 3551 字 | 2.40x |

詩篇は典故密度が高いため散文比約 1.3 倍の補注密度になる。

### Cowork 訳と既訳の比較から学んだ品質特性

| 観点 | Cowork 訳 | 既訳（弘大六 PDF） |
|---|---|---|
| 補注密度 | 多め（10〜25 箇所） | 極小（本文に溶け込ませる） |
| 直訳 vs 意訳 | 直訳寄り | 意訳寄り（典故を踏み込み解釈） |
| 構造忠実度 | 高い（並列構文を保つ） | 並列を脱落させる例あり |
| 伝記整合 | 注意要（在俗時代の事実誤認のリスク） | 概ね正確 |
| 字形読み違え | なし | まれに発生（鳬→かもめ等） |

詳細は引き継ぎメモ各回参照。

### 既訳の発見済みの限界
- **句脱落**：idx=1 老聃・讃毀句の脱落例、idx=0 神泉詩の並列脱落例
- **字形読み違え**：idx=1 鳬→かもめ、麖麞→犛麈の例
- **密教教学用語の通俗化**：idx=1 宝珠→真珠の例
- **並列の脱落**：idx=0 仙詩・神泉詩の二例分割を一例にまとめた例
- **典故誤読**：詳細吟味で発見可

これらは Cowork 訳維持で対応する。

### Cowork 訳の典型誤読（注意点）
- **逐字解釈の取り違え**：idx=1「仁恤は麒鱗に異なり」を「異ならず」と訳した
  例。文意とは反対の誤読が起こりうるので、既訳との突合で見つけて修正する
- **伝記矛盾**：idx=0「絳張にして〜英萃に富めり」を空海在俗時代と齟齬する
  解釈にした例。空海の伝記事実を必ず確認する
- **典故の取り違え**：「吐納＝呼吸」と仙道寄りに解釈する誤読など。文脈で判断

### 原則 13：現代語訳作業中の注追加 OK
注 chu は**辞書的拡張可能フィールド**として運用する。現代語訳作業中に
Cowork が発見した典拠・歴史的背景・字義解釈・関連語彙などは、弘大全集六に
記載がなくても該当篇の chu に追加できる。ただし**現状は補注を本文の括弧書き
内に埋め込み、JSON `chu` には書き込まない**運用（idx=0・idx=1 ともそうした）。
将来 chu 反映時に分離可能。

## 【停止中】自動処理モード（2026-04-24 確立・現在停止中）

ケンシン要望「単純に漢字の入れ替えが目的」を受けて、`_dev_references/process_cycle.py`
を導入。**1 サイクル 1 コマンドで完結**する。判定理由分類（RAW-A/B/C/E/L/O）も
人手の grep 確認も不要。

### 標準フロー（1 サイクル）

```
1. ケンシン：チャットに Gemini OCR 出力を貼付
2. Cowork：貼付を /sessions/.../paste.txt に保存
3. Cowork：python3 _dev_references/process_cycle.py --paste /sessions/.../paste.txt
   （--dry オプションで書込なしの dry-run 可）
4. Cowork：サマリ確認（訂正件数・各篇内訳）して push 依頼
5. ケンシン：commit_push.bat ダブルクリック
6. 次の貼付待ち
```

### process_cycle.py の自動判定ロジック

| chu_check 出力 | 自動処理 |
|---|---|
| 完全一致／語幹一致 | raw のまま |
| VARIANTS 異体字一致 | 書き下し字に置換して raw |
| 1 字違い近接（漢字→漢字）| 書き下し字に置換して raw |
| 1 字違い近接（漢字→ひら/カナ/句読点/活用語尾）| raw のまま（候補棄却）|
| 一致なし | raw のまま |

判定理由分類（RAW-A/B/C/E/L/O）は記録しない。シンプル化を最優先。
省略記法・列挙書式・字義論じ等は「一致なし → raw のまま」で自動的に正しく処理される。

### 篇名解決（自動）

raw 篇頭の【篇名】と JSON `篇名` の差異を吸収するため：
- ひらがな・カタカナ・空白・記号を除外して漢字のみで比較
- VARIANTS テーブルで字形を正規化（灌↔潅、雞↔鷄 等）
- 双方向部分一致＋最長マッチで idx 解決

例：raw「大唐神都青龍寺故三朝国師灌頂の阿闍梨恵果和尚の碑」 → JSON idx=12
「大唐神都青龍寺故三朝の国師潅頂阿闍梨恵果和尚の碑」 自動解決成功。

### raw 篇頭メタの自動形式

```
【JSON 篇名（フル）】
（idx=N・訂正 N 件）

一　見出し　解説…
```

人手で書く長文メタは廃止。

### commit_message.txt の自動形式

```
性霊集 注 raw 追記：idx=N1, idx=N2（訂正 N 件）

idx=N1 篇名: 訂正 X / そのまま Y
  ・注番 旧→新 (raw字↔書き下し字)
idx=N2 篇名: 訂正 X / そのまま Y
```

人手で書く判定理由は廃止。

### Cowork の役割（最小化）

- 貼付保存（heredoc 1 回）
- process_cycle.py 実行（1 コマンド）
- サマリ確認＋push 依頼（1 文）

raw 末尾照合・NULL バイト検証・差分判定・commit_message 執筆は全部スクリプト任せ。
1 サイクル 1〜3 分で完結する想定。

### 例外処理

- 篇名解決失敗 → サマリに「WARN: 篇名「...」未解決・スキップ」と表示

## 【停止中】効率モード運用ルール（2026-04-24 確立・現在停止中）

詩篇は差分が 10 件超えるのが普通。1 篇 1 サイクル＋過剰な文脈確認＋長文
commit_message を続けると、書き下し取込期と同じく時間が無限にかかる。
以下のルールで 1 サイクル 5〜10 分で完結させる。

### ルール E-1：chu_check.py の判定候補をそのまま採用
- 「raw 維持」候補なら raw 維持
- 「書き下し準拠」候補なら FIX-V/D/P で訂正
- 文脈 grep は **「判定候補が明らかに不自然」な時だけ**
- 詩篇では大半が raw 維持。1 件ごとの grep 確認に時間をかけない
- 運用 4 ルール第 3 条「判定候補そのまま採用 OK・境界例で悩まない」徹底

### ルール E-2：複数篇まとめ処理
1 貼付に複数篇含まれる場合、必ず 1 サイクルで完結させる：

```python
# 1) 篇分割（python re.split で 【篇名】区切り）
# 2) for ループで chu_check.py を篇ごとに連続実行
# 3) 判定一括 → raw 追記 1 回 → commit_message.txt 更新 1 回 → push 依頼
```

篇ごとに別サイクルにしない（オーバーヘッドが致命的）。

### ルール E-3：raw 篇頭メタは最小限

**必須**：日付不要・`（idx=N・書き下し準拠 N 件・raw 維持 N 件）`
**任意**：raw 維持件数が多いとき、1 行 1 件で簡潔に判定コード併記
**省略可**：過去サイクルのような長文の傍証パターン詳細列挙

例（詩篇典型）：
```
【篇名 詩型】
（idx=N・書き下し準拠 0 件・raw 維持 3 件）

一　見出し　解説…
```

### ルール E-4：commit_message.txt は 10〜25 行

構成（最大 5 セクション）：
1. 1 行目：取込範囲＋件数サマリ
2. ■ 取込範囲（idx ごとに 1 行）
3. ■ 突合：一致 N / 差分 N / 字形訂正 N / raw 維持 N
4. ■ 字形訂正（あれば 1 行 1 件）
5. ■ raw 維持（1 行 1 件・判定コード併記）
6. ■ 備考（順序ずれ・VARIANTS 候補があれば）

過去サイクルのような長大なメタ説明は不要。

### ルール E-5：差分判定は機械的に（判定フローチャート）

| パターン | 判定コード | 対応 |
|---------|-----------|------|
| 見出し末尾「…」「…（以下〜）」 | RAW-E | raw 維持（省略記法）|
| 見出し「X・Y」形＋両字書き下しにあり | RAW-L | raw 維持（列挙書式）|
| 解説文「X は…の意」「X、〜也」 | RAW-A | raw 維持（字義論じ）|
| 解説文「諸本では」「勝又本」「誤字か」 | RAW-B | raw 維持（底本差）|
| 解説文の典故引用『〜』内に対象字 | RAW-C | raw 維持（典故引用）|
| VARIANTS 異体字＋解説文傍証なし | FIX-V | 書き下し準拠で訂正 |
| 踊り字（営営↔営々） | FIX-D | 書き下し準拠で訂正 |
| 助詞・句読点差（、↔て） | FIX-P | 書き下し準拠で訂正 |
| 上記いずれにも該当せず | 候補そのまま | chu_check の判定に従う |

境界例（idx=3 注七 解頤型：字句両方ありで配列差）は raw 維持で OK。
深く考えない。

### ルール E-6：1 サイクルの bash チェーン

理想形（1 つの bash 実行ですべて完結）：

```bash
# Step 1: 貼付保存＋篇分割
cat > /sessions/.../paste.txt << 'PEOF' ... PEOF
```

## git 運用

### 書き込み系 git は .bat 経由（必須）
Cowork サンドボックスから `.git/` 内のファイルを操作できません
（Operation not permitted）。そのため commit / push / reset / stash / checkout
などの書き込み系 git 操作は、必ず Windows 側で .bat ダブルクリックで実行
します。サンドボックスからの git 実行は **読み取り系のみ**（log, status,
diff, show）に留める。

### .bat 内の日本語は禁止・必ず外部ファイル経由
cmd.exe は既定で Shift-JIS で .bat を読むため、UTF-8 で書かれた .bat に
日本語を含めるとコマンドが誤解釈されて実行エラーになります。

**必ず以下の形にする**：
- .bat の中身は ASCII・英語のみ
- 日本語のコミットメッセージは `commit_message.txt`（UTF-8）に書く
- .bat からは `git commit -F commit_message.txt` で読ませる

### JSON/raw 書き込み後の NULL バイト検証
サンドボックスから JSON や raw を上書きする際、元ファイルが新ファイルより
大きいと truncate が効かず末尾に NULL バイトが残ることがあります。
書き込み後は必ず：

```python
with open(path, 'rb') as f:
    raw = f.read()
print('trailing nulls:', len(raw) - len(raw.rstrip(b'\x00')))
# 0 でなければ破損
```

破損した場合は `git show HEAD:<path> > <path>` で復旧する。

### Windows マウント上のファイル削除
`rm` コマンドは Operation not permitted で失敗する。削除が必要なときは
Windows 側で `del /f /q` を使う（.bat 経由）。

### 定番 .bat（リポジトリ直下）
- **`commit_push.bat`** … メイン運用（commit + push を 1 ダブルクリックで完結）
- `1_commit.bat` … commit のみ
- `2_push.bat` … push のみ
- `verify.bat` … git log + status 確認
- `fix_index.bat` … git index 破損回復

### commit_push.bat の固定 stage 対象（5 ファイル）
毎回書き換えず以下を固定列挙：

```
git add data/kukai/shoryoshu_miyasaka.json
git add _dev_references/shoryoshu_vol5_kindle_ruby.md
git add _dev_references/kindle_ocr_rules.md

```

変更のないファイルに `git add` しても no-op なので害なし。

- `_archive/scripts/` … 過去の .bat（リセット作業用など）
- `_archive/memos/` … 過去の引き継ぎメモ・書き下し期 CLAUDE.md

日常作業では触らない。緊急時の参考資料として保全。

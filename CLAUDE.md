# buddhist-data-api 作業ルール
このリポジトリは、空海と真言密教の知識を蓄積する「共有知識バンク」です。
新しいCoworkセッションでこのリポジトリの作業を始めるときは、必ず最初に
このファイルを読んでから作業を始めてください。
---
## 基本方針
### 共有知識バンクであること
このリポジトリは、kaimyo-app(戒名・諷誦文・法話生成アプリ)だけのための
データベースではありません。将来、複数の仏教関連アプリで共有して使うことを
想定しています。例えば:
- 戒名・諷誦文・葬儀法話の生成アプリ
- 葬儀以外の日常の法話生成アプリ
- 仏教関連のブログ制作アプリ
- 真言宗教義の勉強会・学習用アプリ
- その他、これから作るかもしれない仏教関連アプリ
これらのアプリ構想はまだ初期段階ですが、データベース自体は今から
複数用途に対応できる形で蓄積していきます。
### データ蓄積の方針
- 特定のアプリ用に最適化したデータ抽出はしないこと
- 「これは戒名作成には使えないから入れなくていい」のような判断をしないこと
- できるだけ豊かに、できるだけ網羅的にデータを保存すること
- 葬儀には使えなくても、日常の法話やブログ、教義学習で価値がある内容も
  すべて取り込むこと
- 空海の言葉と真言密教の知識を、可能な限り完全な形で蓄積すること

## データ抽出と校注作業のやり方
### 最重要原則:書き下しテキストデータが絶対の基準
ユーザー(ケンシン)と一緒に作成した書き下しテキストデータ
(`_dev_references/shoryoshu_kakikudashi.txt` 等)は、信頼できる正確な典拠です。
Coworkの仕事は、このテキストデータを忠実に参照してJSONに反映することです。
優先順位は絶対に: 書き下しテキストデータ > Coworkの知識
テキストデータと異なる内容をCoworkが勝手に書いてはいけません。
Coworkの知識は、テキストデータでカバーされていない箇所を埋めるための
「補助」としてのみ使います。
### 作業の流れ
1. 書き下しテキストデータを参照する
   - これが第一の作業。テキストデータの内容が絶対の基準。
   - PDFの直接読み取りや SAT照合は基本的には行いません。
2. テキストデータに記載のある部分は、そのまま採用する
   - Coworkの知識で「もっと良い表現があるはず」などと
     勝手に書き換えてはいけない。
   - テキストデータに書いてあることが、そのまま正しい内容。
3. テキストデータでカバーされていない部分のみ、Coworkの知識で補完する
   - 該当箇所がテキストデータに含まれていないとき、
     初めてCowork自身が持っている空海の著作の知識を使う。
   - その知識を使って「筋が通るように」補完する。
   - これはあくまで補助的な作業であり、テキストデータ内の記述を
     上書きしてはならない。
4. 判断に迷ったときは、必ずユーザーに確認する
   - 「テキストデータのここに該当箇所が見当たらない、Coworkの知識では
     こう推測するが確認してほしい」と報告する。
   - 勝手に判断して進めないこと。
5. 修正済みのテキストを data/kukai/ の該当するJSONファイルに追加する。
6. コミットする。
7. push.bat を手動実行してGitHubに反映する(ユーザーが実行)。

### ⚠ shoryoshu_kakikudashi.txt の巻5-10 列混在問題（2026-04-18 判明）
`_dev_references/shoryoshu_kakikudashi.txt` の **巻第五以降** は、
Kindle 宮坂版の多段組レイアウトに対して fix_columns.py がうまく効いておらず、
本文の行順が列ごとに混ざっています。例：

```
伏して惟れば動止万福…誰か腸          ← 行末で次の列へ
道豈に息まんや。賀能…仲春漸く暄か     ← 別の列
渤海と日本と地南北に分れ…往古今来斯の
を断つに堪えん。今日別を取る…不宜謹んで状す
```

正しい読み順：L3→L2→L1→L4 のように並び替えが必要。

このため、**巻第五〜第十のJSONへの取り込みは、書き下しファイルから直接
コピーせず、ケンシンに Kindle から clean 版を送ってもらう運用** に切り替えた
（2026-04-18）。過去の巻第一〜第四は旧セッション（Opus 4.6）がケンシンの
Kindle OCR 貼り付けと組み合わせて乗り切った履歴あり。

巻第一〜第四の既存JSONデータは無事。巻第五以降を作業する際は必ずこの
運用ルールに従うこと。

#### 追記（2026-04-20）：Kindle アプリの画面表示自体にも列ズレが発生する

これまで「列混在問題」は fix_columns.py（テキスト抽出）側の問題と理解
してきたが、2026-04-20 に **Kindle アプリの画面表示そのもの** にも同種の
列ズレが発生することが判明した。

具体例（idx=39「越州の節度使に与えて内外の経書を求むる啓」末尾の
スクショ）:

- Kindle 表示の行4末: 「遠方に流伝せしめよ。儻渇」
- Kindle 表示の行5: 「ち大士の経営する所、小人の意わざる所なり。斯れ則」
- Kindle 表示の行6: 「仰を遂げば茂績英声…」

→ 正解順序: 「遠方に流伝せしめよ。**斯れ則**ち大士の経営する所、小人の
意わざる所なり。**儻渇**仰を遂げば茂績英声…」

つまり、行4末の「儻渇」と行5末の「斯れ則」（各2-3字ブロック）が
Kindle 表示上で **入れ替わって** 表示されていた。これは fix_columns.py の
問題ではなく、**Kindle 配信データ（XHTML/XML）レベルでブロック順序が
既に誤っている可能性** が高い（だから kakikudashi.txt でも Kindle 表示
でも同じ列混在が出る）。

**対策（再発防止）**:
1. Kindle スクショから OCR した後、必ず読み下しとして意味が通るか
   流し読みチェックする
2. 漢文訓読として不成立な繋ぎ（例：「渇ち大士」「則仰を遂げば」など）に
   気付いたら、推測で並び替えず必ずケンシンに相談して正解順序を確定
3. 中間ファイル（`_dev_references/shoryoshu_vol5_kindle_ruby.md`）には
   「Kindle 表示上は列ズレ有、正解順序はケンシン目視確定」と注記を残す
4. 「見たまま再現」原則1 は字形・ルビには適用するが、**Kindle 表示自体が
   壊れている場合の行順** は対象外（意味的整合チェックが優先）

## 守るべき原則
1. 既存のJSONファイルの構造を勝手に変更しないこと
   - 必ず最新のコミット内容を確認してから、同じ形式で続けること
   - 古いやり方に戻らないこと(作業のやり方は段階的に進化してきている)
2. 典拠を必ず明示すること(空海の精神に従う)
   - どの願文・どの章から引用したかを必ず括弧書きで示す
   - 空海自身が経典の典拠を尊重した態度を、データ作成にも継承する
3. 「賀」と「訶」のような意図的な使い分けは保持すること
   - 真言宗の基本経典では「訶」が通常訳
   - ただし空海『吽字義』では『理趣釈経』を引用するときだけ「賀」を使う
   - これは空海が引用元の表記を尊重した結果であり、機械的に統一しない
4. AIが勝手に作り出した言葉は使わないこと
   - 必ず書き下しテキストデータの内容と、Coworkが持っている空海の著作の
     正確な知識に基づくこと
5. エラーが出たら勝手にリトライせず、必ず報告すること
6. 一度に大量に処理せず、章ごと・段階的に進めること
   - コンテキスト溢れを防ぐため
7. data/ の中身と _dev_references/ の中身を混同しないこと
8. Cowork が書き下しファイルの断片を黙って再構築して JSON に入れることを禁止する
   - 推測による再構築が必要な状況になったときは必ずケンシンに報告し、
     Kindle から clean 版を送ってもらう運用に切り替える
   - 推測で進めて【要確認】マークも付けずに黙ってコミットする、のは厳禁
9. 目次と本文題にズレがあっても、修正せずにそのまま記録する（2026-04-19 追記）
   - 古い本では目次と本文の篇題が一致しないことがある
   - どちらが「正しい」かを Cowork の判断で書き換えてはいけない
   - 目次表記・本文表記どちらも原文通り残す。必要に応じて口伝（kuden）や
     異同として CLAUDE.md または chu に記録する
   - 例：巻第六の #2「天長皇帝故中務卿の親王の為に…」
     - 目次: 「法華経を講ずる願文」
     - 本文: 「田及び道場の支具を捨てて橘寺に入るる願文」
     → どちらも原文通り残す
10. Gemini OCR の「勝手な手直し」に注意（2026-04-19 追記）
    - Gemini は OCR 結果を「読みやすい常用漢字」に自動変換することがある
      （例：椎→権/榑、澍→潤、戚→咸、髪論→髣髴、放昿→放光）
    - Kindle 原文と照合して、レア漢字・古字は Kindle の表記を採用する
    - Cowork 自身のビジョン読解もハルシネーションしうるので、
      最終確定は必ずケンシンの目視チェックを経ること
11. ラベルより内容を信頼すること（2026-04-20 追記）
    - 引き継ぎメモが「idx=N の続き」「特定ファイルの修正」などラベル参照で
      位置を指しているときは、必ずメモ記載の本文の特徴的な一節を `grep` で
      repo 内検索して、ラベルと内容が同じ場所を指しているか逆引き確認する
    - ラベル（idx 番号・ファイル名・コミットハッシュ・ページ番号）は人間が
      書く情報なので誤記が混入しうる。内容（本文の文字列）は repo 上の
      客観事実なので、両者が食い違ったら**内容を信頼しケンシンに報告**する
    - 例（2026-04-20 実際に発生）：メモに「idx=37 の続き、邇《と》め 確定」
      と書かれていたが、邇 を含む本文は JSON では idx=39 に存在していた。
      新セッション開始時に内容照合しなかったため、OCR 転記後に差分照合で
      初めてズレに気付いた事故。Step 3・4（下記「新しいセッションを始める
      ときの手順」）を最初に実行していれば事前検出できた

## git push のタイミング（2026-04-21 再改訂：スクショ作業ごとに 1 サイクル）

### push タイミングのルール（新運用）
**スクショ作業 1 単位ごとに、JSON 反映 → commit → push までを 1 サイクルで完結**
させる。中間ファイル反映だけで終わらせない。

具体的には以下の 1 サイクル：

1. ケンシンがスクショ送信
2. Cowork が OCR（①ルビ付き本文・②新規確定ルビ・③不確定点）
3. ケンシン確認（問題なし or 訂正指示）
4. Cowork が即座に中間ファイル更新・JSON 反映・NULL バイト検証・
   commit_message.txt 更新
5. **ケンシンが `1_commit.bat` → `2_push.bat` を実行**
6. push 成功確認後、次のスクショへ

### なぜこのルールに変えたか（2026-04-21 idx=37, 38 削除事故を受けて）
- 旧運用（2026-04-20 改訂版）では「commit は作業区切りごと・push はケンシン
  明示指示 or 引き継ぎ時」としていた
- しかし「中間ファイルに反映済み → JSON 未反映の状態が長く続く」リスクが
  残っていた
- 2026-04-19 に idx=37, 38 で Kindle 清書版を JSON に入れたあと、2026-04-20 に
  「旧書き下し全削除」の意図で削除した際、**ケンシン自身がその削除を後に
  失念**。2026-04-21 セッションで Cowork が「未取得」と報告してようやく
  ズレが発覚した
- スクショごとに即 push すれば：
  - ローカル・origin 双方に常に最新版が存在 → 復元容易
  - 進捗表と実態のラベル vs 内容不一致リスクが極小化
  - 中間ファイル・JSON・git 履歴の 3 者が常に同期
- 粒度が細かすぎて commit 数が多くなる懸念はあるが、**事故防止のメリットが
  commit 履歴のノイズコストを上回る**と判断

### 旧ルールの変遷（参考）
- 旧々（〜2026-04-19）：20 ページごとに push
- 旧（2026-04-20）：ケンシン明示指示 or 引き継ぎ時のみ push
- 新（2026-04-21）：スクショ作業ごとに commit+push 1 サイクル ← **現行**

### commit_push.bat を新設（2026-04-21）
新運用の標準 `.bat` として `commit_push.bat` をリポジトリ直下に追加。
1 ダブルクリックで **stage → commit → push → verify** まで完結。

- Step 1〜6 は旧 `1_commit.bat` と同じ（ロック削除・reset・stage・diff 表示
  ・commit_message.txt 確認・commit）
- Step 7 で `git push origin main`
- Step 8 で verify（`git log --oneline -3`・`git status`・
  `git log origin/main..HEAD` で sync 確認）
- commit 失敗時は push をスキップ（errorlevel で goto :end）

### 1_commit.bat・2_push.bat の扱い（2026-04-21 以降）
- 通常運用は `commit_push.bat` 一本
- `1_commit.bat`（commit のみ）・`2_push.bat`（push のみ）は、
  **commit のみ / push のみが必要な特殊ケース** のために残す
  （例：commit だけ先に積みたい、既に commit 済で push だけ再試行したい）
- 中身は 3 ファイル固定 stage に統一済

### 1_commit.bat の汎用化（2026-04-21）
Step 3 の `git add` は以下 3 ファイル固定：

```
git add data/kukai/shoryoshu_miyasaka.json
git add _dev_references/shoryoshu_vol5_kindle_ruby.md
git add CLAUDE.md
```

変更のないファイルに `git add` しても no-op なので害なし。毎回 `.bat` を
書き換えると事故の元（2026-04-21 のケース：CLAUDE.md 追補後のまま放置
されて JSON が stage されず `git commit failed` になった）なので、
汎用化した。

### 連続実行のルール
スクショごとのサイクルは、ケンシンのスクショ受領・確認・`.bat` 実行が挟まる
ので Cowork 側の連続実行はなし。Cowork 側は「確認 → 反映 → commit_message 更新
 → ケンシンに `commit_push.bat` 依頼」で待機する。

### 途中で止まってよい例外
- テキストデータでカバーされていない箇所が多発して【要確認】が連続する場合
- テキストデータの構造が想定と違って判断に迷う場合
- コンテキスト上限が近いと自己申告する場合
- エラーが出た場合

上記以外は途中報告なしで連続実行する。

## 技術的ノウハウ（2026-04-17 追記・Opus 4.7 以降）

2026-04-17 に Opus 4.7 へアップデートされ、同時期に Cowork のインフラ挙動が
変わったことで、以下の技術的な落とし穴が発覚した。新セッションで git や
ファイル操作を行うときは、必ず以下のルールに従うこと。

### 1. git のコミット・リセット・stash 操作は必ず Windows の `.bat` から実行する
Cowork サンドボックスからは、Windows マウント上の `.git/` 内のファイルを
削除できない（Operation not permitted）。そのため、stash、reset、lock ファイル
削除などを伴う git 操作は、必ず Windows 側で `.bat` ファイルをダブルクリック
して実行する。サンドボックスからの git 実行は読み取り系（log, status, diff,
show）のみに留めること。

### 2. `.bat` ファイル内の日本語メッセージは必ず外部ファイル経由で渡す
cmd.exe は既定で Shift-JIS で `.bat` を読む。UTF-8 で書かれた `.bat` の中に
日本語を含めるとコマンドが誤解釈されて実行エラーになる（例：`set MSG=...p.299...`
が `'299' は、内部コマンドまたは外部コマンド…` として失敗する）。

**必ず以下の形にする：**
- `.bat` の中身は ASCII・英語のみ
- 日本語のコミットメッセージは `commit_message.txt`（UTF-8）に書く
- `.bat` からは `git commit -F commit_message.txt` で読ませる

### 3. JSON 書き込み後は必ず末尾 NULL バイトを検証する
Cowork サンドボックスから JSON ファイルを上書きする際、元ファイルが新ファイル
より大きいと truncate が効かず末尾に NULL バイトが残ることがある。書き込み後
は必ず以下を確認：

```python
with open(path, 'rb') as f:
    raw = f.read()
print('trailing nulls:', len(raw) - len(raw.rstrip(b'\x00')))
# 0 でなければ破損している
```

破損した場合は `git show HEAD:<path> > <path>` で復旧する。

### 4. Windows マウント上のファイルは Cowork サンドボックス側で削除しない
`rm` コマンドは Operation not permitted で失敗する。削除が必要なときは Windows
側で `del /f /q` を使う（`.bat` 経由）。

## 新しいセッションを始めるときの手順

新しい Cowork セッションでこのリポジトリの作業を始めるときは、以下の手順を

### Step 1: 自動読み込み
1. この CLAUDE.md を最初に読む（自動で読まれる）
2. リポジトリ直下の `引き継ぎメモ_*.md` のうち最新のものを読む

### Step 2: repo 状態の確認（git 読み取り系のみ）

```
git log --oneline -5           # 直近 5 commit
git status --short             # 未コミット差分
git log origin/main..HEAD      # 未 push commit
```

CLAUDE.md 技術ノウハウ 1 に従い、**書き込み系 git 操作はここでは実行しない**
（Windows 側の `.bat` 経由で行う）。

### Step 3: 引き継ぎメモの主張を内容で検証する（必須・最重要）

引き継ぎメモは人間が書く文書のため、ラベル（idx 番号・ファイル名・コミット
ハッシュ・ページ番号）の誤記が混入しうる。**作業を開始する前に、引き継ぎ
メモに書かれているすべての主張を repo の実状態と照合すること**。

次のパターンは必ず検証する：

| メモの主張 | 検証コマンド |
|-----------|--------------|
| 「CLAUDE.md を commit 済」「未コミット」 | `git log -1 -- CLAUDE.md` で hash 確認 |
| 「JSON の idx=N にこの本文がある」 | `grep "特徴的な一節" data/kukai/*.json` で idx と本文を突き合わせる |
| 「commit_message.txt 更新済」 | `head -1 commit_message.txt` で先頭行確認 |
| 「ケンシン確定字は◯」 | JSON を grep して現在の字と確定字が一致するか |
| 「未コミット差分あり」 | `git diff --stat` と `git status --short` で実差分を確認 |

### Step 4: ラベル参照と内容参照の整合性確認（守るべき原則 11）

引き継ぎメモが「idx=N の続き」「特定の本文の修正」のような **ラベル参照**
で位置を指しているときは、メモに載っている本文の特徴的な一節を `grep` して
**idx 番号と本文が実際に一致するか確認**する。

ズレていたら **作業を始めずケンシンに報告**（原則 11）。

### Step 5: 作業開始

Step 3・4 で齟齬がないことを確認したら、本 CLAUDE.md の「データ抽出と校注
作業のやり方」に従って続きを進める。

齟齬があった場合の対応：

- ラベルと内容の不一致 → ケンシンに報告、どちらが正しいか確認
- 「未コミット」と書いてあるが実際は commit 済 → 該当タスクはスキップ
- 「ここまでやった」と書いてあるが実際は未実施 → ケンシンに再確認

### 参考：過去の事故（2026-04-20 idx=39 誤認事件）

引き継ぎメモに「idx=37 の続き・邇《と》め 確定」と書かれていたが、邇 を

ズレに気付いた。Step 3・4 を最初に実行していれば OCR 転記前の時点で検出

## 巻第五 Kindle スクショ OCR → JSON 反映の作業フロー（2026-04-19 確立）

巻第五以降は kakikudashi.txt が列混在で使えないため、ケンシンが Kindle
から直接スクショを送り、Cowork が OCR してルビ付き清書版を作る運用。

### 原則（これを守らないとケンシンが疲弊する）
1. **見たままを再現する**。Cowork は解釈を挟まない。文脈で「こうあるべき」
   という補正も禁止。字形・ルビ位置・漢字はスクショで見えた通りに書く。
2. **ルビ位置が2字にまたがって曖昧なときも、見たままを書く**。親字を動かさない。
   「この漢字にこのルビは意味的におかしい」と思っても書き換えない。
   訂正はケンシンが目視で指摘する。ケンシンの指摘は絶対の修正指示。
3. **判別できない字は「?」マークを付けて保留**。推測で埋めない。
   例：`膝歩《しつほ?》` のように `?` 付きで書き、拡大スクショを待つ。
4. **表示形式**：
   - ① ルビ付き本文（《》後置形式、例：`膝歩《しつほ》`）
   - ② ルビだけ一覧
   - 修正指示が来たら、**該当部分だけ再表示**。毎回全文を出すとケンシンが
     確認しにくい。
5. **Cowork 自身の知識で補完しない**。ケンシン確定事項（Kindle ページ番号
   指定や拡大スクショ）だけが正しい根拠。

### ルビ形式
- 親字の直後に `《》` で後置。例：`膝歩《しつほ》`、`戕風《しょうふうかじ》`
- 2字以上にまたがるルビは親字をまとめて1単位として扱う
  （例：`迷霧氛氳《めいむふんうん》`）
- 旧字・異体字（棆・㠊・屩・墀 等）は Kindle 原字を維持。常用漢字に直さない。
  Gemini OCR は自動補正する癖があるので Kindle と照合（CLAUDE.md 原則10）。

### 作業ステップ
1. **ケンシンから Kindle スクショを受領**（1篇あたり複数枚）
2. **スクショごとに ① ルビ付き本文＋② ルビだけ一覧 を表示**して確認を受ける
3. **確認を受けた都度、中間ファイル `_dev_references/shoryoshu_vol5_kindle_ruby.md`
   に追記保存**（JSON には即書き込まない。中間で一段落させる）
4. **未取得部分は【GAP: ...】マーカーで明記**。ページ境界は CLAUDE.md 原則9
   に従って書き換えず、Kindle 原文のまま残す。
5. **全スクショが揃った時点で JSON `data/kukai/shoryoshu_miyasaka.json` に反映**
   - 単一ページ篇（page_number=None のもの）は page[0].kakikudashi に直接書く
   - 複数ページ篇（idx=37 のような 5 ページ構成）は既存の page_number ごとの
     区切りを維持し、ページ境界で split して書き込む
   - 【GAP】マーカーは kakikudashi フィールドに残したまま JSON に書き込む
     （後日埋めることで GAP が消える）
   - **【要確認】は editorial note なので JSON には入れない**。中間ファイルにのみ残す。
6. **NULL バイト検証**（CLAUDE.md 技術ノウハウ 3 に従う）
7. **commit_message.txt を更新**してケンシンに `1_commit.bat` を依頼

### 中間ファイル（`_dev_references/shoryoshu_vol5_kindle_ruby.md`）の構造
- 篇ごとに `## idx=N　篇名` ヘッダ
- 各セクション:
  - `ページ:` 行（暫定でも記録）
  - `**状態**:` 行（OCR 完了状況）
  - 本文（○篇題に続けて書き下し）
  - 【GAP: ...】マーカー（未取得部分）
  - 【要確認: ...】マーカー（ケンシン確認待ち項目）
- セクション末尾に `---` 区切り

### 原則の運用上の注意（2026-04-20 追記：原則1の拡充・再発防止）

これまでに発覚した原則違反と、再発防止の運用を以下に明文化する。今後のセッションではこの節を必ず先に読み、出力前の自己チェックに使うこと。

#### 発覚した違反例（2026-04-20）

- **親字を意味で動かした**：ルビ「あと」がスクショ上で「武」の横にあるのに、Cowork が「武に『あと』は変だから『興る』の誤植だろう」と判断し、「武を継ぎ、聖帝重ねて興《あと》る」と親字を動かして出力した。
- **ルビの清濁を意味で補正した**：ルビが「けいたい」（清音）なのに、「驚汰」を「鯨汰（げいた）」と意味解釈したい誘惑に引きずられ、「げいたい」と勝手に濁点を足した。
- **字形を似た字に補正した**：「刳《くぼ》めて」を「剝《くぼ》めて」、「銜《ふく》み」を「徇《ふく》み」と偏旁の違いを見落として自分の語彙に寄せた。
- **ルビの拾い落とし**：「膽《たましい》」のルビを親字横にあるのに見落とした。
- **Gemini OCR 補正を見破れず採用**：昨日の OCR で「瀆」を「黷」、「法の流塞只だ吐納に繋がれり」を「観音の一身付属の四衣なりと」と誤変換した結果をそのまま JSON に書き込んだ。
- **同じスクショ内で連続字形誤認（2段階ミス）**：idx=37 の「邇《と》め」を最初「遍《と》め」と誤出力。ケンシンから「遍も同じ間違いか調べて」の指摘を受けた直後、**拡大画像を見せられた上で**「訪《と》め」と再び誤推測した（しかも辶が明確に見えているのに言偏と言い張った）。潅→涟→澍 の3連続ミスの直後に同じパターンを再発。これは「ケンシンの訂正指示が来た＝字が違うという情報」を「Cowork 自身が次の候補を当てていい」と勘違いしたことが根因。
- **字形確認で輪郭パターンマッチに頼った**：潅・邇 とも偏旁を1画ずつ分解せず、「なんとなく似た字」に寄せた。拡大画像があっても構造分解を省略すれば同じ誤りが起こる（邇の時は拡大画像を渡されていた）。
- **口伝読みを辞書読みで否定しようとした**：邇《と》の「と」は口伝読みだが、辞書に載っていないという理由で「ルビ読みとして自然な字（訪・覓など）に親字を動かしたい」誘惑に引きずられた。これは CLAUDE.md「口伝前提の原則」への正面違反。

#### 発覚した違反例（2026-04-21）

- **同一セッション内で字形輪郭マッチ癖が 3 連続再発**：
  1. idx=43「屩を躧いて」→正は「屩を躡いて」（足偏＋麗 と 足偏＋聶 を偏旁分解せず誤読）

  3. idx=41「鉛素に縟んずべし」→正は「鈆素」（金＋㕣 と 金＋公 を偏旁分解せず誤読）

  → CLAUDE.md 再発防止 5（2026-04-20 追加済）を**宣言だけで実行できていない**。
    同一スクショの中で 3 連続同パターンの失敗が起こる以上、「心がけ」では足りず、
    **出力直前に偏旁分解の結果を明示的に書き出す手順**を実作業に組み込む必要がある
    （再発防止 9 として新規追加、下記）。

- **ルビの完全拾い落とし（親字にルビがあるのに「ルビなし」と処理）**：
  - idx=41「且《かつか》つ」を「且つ」と出力。親字「且」に 3 拍連語ルビ「かつか」が

#### 「見たまま」の定義（拡充）

「見たまま再現」は以下すべてを含む。どれか1つでも崩すと原則1違反となる。

- **親字の字形**：偏旁の違い（木偏と忄偏、刂と剝、金偏と彳偏 等）を勝手に補正しない。自分の知っている近い字に寄せない。
- **親字の位置**：ルビが A 字の横にあるのを、意味で通りそうな B 字に動かさない。
- **ルビの清濁・半濁**：「け」を「げ」、「は」を「ば・ぱ」と勝手に変えない。
- **ルビの長音・拗音・促音**：「ゅ」を「ゆ」、「っ」を「つ」、「ー」を「う」などに勝手に変えない。
- **ルビの有無**：親字ごとにルビがあるかを1字ずつ確認。拾い落とし禁止。

#### 口伝前提の原則

性霊集をはじめとする宗教書・古典は、字面の常識的な読みではなく**口伝での読みが正**である。「武《あと》」のように辞書的には意味が通らないルビも、口伝では正しい。「意味が合わない」と感じるのは Cowork の知識不足であり、テキストの誤りではない。違和感を根拠に親字もルビも動かしてはならない。

#### Kindle 底本の原本写し間違い

宮坂版 Kindle の底本には、原本そのものの写し間違いが含まれる字がある（例：正しくは「凱風」が「飑風」と印刷されている）。この種の誤りは Kindle を見たまま再現しても誤ったままになる。**ケンシンの口伝確定指示が Kindle 表記より優先**する。ただしこの判定は Cowork 側ではできないので、違和感があれば訂正せず質問・保留する。

#### 再発防止の運用（出力前チェックリスト）

1. **スクショ読解を2段階に分離**。
   - 第1段階：親字のすぐ横にあるルビを視覚情報だけで拾う。意味判定は一切しない。
   - 第2段階：出力前に「親字を意味で動かしていないか／ルビの清濁・拗促音を意味で動かしていないか／ルビ拾い落としがないか」を自己チェック。
2. **違和感を感じたら訂正しない**。「?」保留か、ケンシンに質問する。
3. **訂正はケンシンの目視指摘のみが根拠**。Cowork の知識・推測は根拠にならない。
4. **ルビ範囲の曖昧さ**（1字かかるか2字にまたがるか）だけは従来通り「連語ルビ（上下まとめて1単位）」で表示し、「?」は付けない（2026-04-20 ケンシン確認）。
5. **字形確認は「偏旁を1画ずつ分けて描画する」作業として行う**。輪郭のパターンマッチは禁止。拡大画像が渡されても構造分解を省略すれば誤りが止まらない（潅・邇の実例）。具体的には：左側の偏／右側の旁／上部／下部／内部 をそれぞれ独立に画単位で列挙してから字を合成する。
6. **ケンシンの訂正指示が来た後、推測で次の候補を出さない**。訂正指示は「その字は違う」という情報であり、「正しい字はこれ」という情報ではない。正しい字はケンシンから明示されるまで**必ず待つ**。「では次は◯◯でしょうか？」と連続推測するのは、潅→涟→澍／遍→訪→ の再発パターン。
7. **ルビ付き字とルビなし字で注意の濃度を変えない**。ルビがないからといって字形確認を省略しない（涟→潅の誤読はルビなし字で発生）。
8. **口伝読みに辞書読みをぶつけない**。「このルビは辞書に載っていないから、別の親字が正しいのでは」という発想自体が原則違反。辞書読みと一致しない＝口伝読みである可能性が高い、というのが性霊集の標準。
9. **偏旁分解を出力前に明示的に書き出す（2026-04-21 追加）**。漢字ごとに「左の偏／右の旁／上部／下部／内部」を独立に列挙し、手元のメモとして書き出してから字を合成する。輪郭だけでパターンマッチしない。具体的な失敗例：
   - 躧（足＋麗）と躡（足＋聶）：麗 ≠ 聶 を偏旁分解で切り分ければ即判別可能
   - 納（糸偏＋内）と衲（衤偏＋内）：糸偏と衤偏は偏の 1 画目と下部の形で区別可能
   - 鉛（金＋㕣）と鈆（金＋公）：㕣 は「八＋口」、公 は「八＋厶」で下部が違う

   これらは全て「偏」または「旁」の構造を 1 画ずつ見れば迷わない違い。ところが
   輪郭パターンマッチに頼ると「なんとなく似た字」で出力してしまう。**再発防止 5
   の運用版**：スクショの字を書き下し出力する前に、内部的に偏旁を分解した旨を
   出力に盛り込む（「◯（偏＝X、旁＝Y）」とメモ書きしてから字を確定する）。

10. **親字ごとのルビ有無を 1 字単位で個別確認する（2026-04-21 追加・同日強化）**。行全体を眺めて
    ルビを拾うと、目立つルビだけが目に入り、「且《かつか》つ」のように親字にルビが
    付いているのを完全にスキップする失敗が起きる。**親字 N 個ある行では N 回、
    「この字にルビあり／なし」を親字ごとに個別判定する**。ルビなしと判定したら
    1 秒立ち止まって「本当に横に何もないか？」と再確認する。

    **（2026-04-21 強化：選択的列挙の禁止）** idx=37「逐《したが》い」拾い落とし事故
    を受けて、「個別確認」は宣言だけでは足りず **全漢字を漏れなく列挙する手順**
    を強制する。具体的には：
    - 出力前に、本文中の **全ての非ひらがな文字（漢字）** を出現順に列挙する
      内部メモを作る（例：`能・憚・献・誠・是・逐・赴・明・身・知・然・猶・命・…`）
    - 列挙した **1 字ずつに対して**「ルビあり《...》」か「ルビなし」を明記
    - 列挙から抜けている漢字が 1 字でもあれば、その時点で確認は不完全とみなす
    - 「11 字だけ見て大丈夫そう」は禁止。**部分確認は未確認と同じ**扱いにする
    - 拾い落としが発生した場合、原因は必ず「列挙の不完全」と断定する（注意力
      不足や運ではなく、手順の欠落）

    選択的確認に戻るのが一番危険なパターン。ルール 10 は「何文字あっても例外なく
    全部列挙する」として運用する。

### 今回セッション（2026-04-21）で確立した作業リズム

Kindle スクショ受領 → OCR → ケンシン確認 → 中間ファイル反映 の流れが安定した。

1. **ケンシンからスクショ受領**（1〜数枚まとめて）
2. **スクショごとに出力**：

3. **ケンシン確認**：
   - 「問題なし」→ そのまま中間ファイルに反映し、次のスクショ受領に進む
   - 「X→Y に訂正」→ その箇所だけ反映し、次のスクショ受領に進む
     （**再表示は該当部分のみ**。全文を出し直さない。ケンシンの確認負担を減らす）
4. **中間ファイルへの反映**（`_dev_references/shoryoshu_vol5_kindle_ruby.md`）：
   - 該当 idx セクションの本文を書き換え
   - 【GAP】マーカーの位置を前進
   - **状態** 行を更新
   - 最終更新行を追記（新規確定ルビ・訂正点を明記）
5. **JSON 反映は本文完結時点で行う**。中途では入れない（巻第五作業フロー 5 に準拠）。
6. **commit は本文完結または区切り時点で `1_commit.bat` 依頼**。

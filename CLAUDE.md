# buddhist-data-api 作業ルール（注 chu 作業専用・2026-04-24〜）

このリポジトリは、空海と真言密教の知識を蓄積する「共有知識バンク」です。
新しい Cowork セッションでこのリポジトリの作業を始めるときは、必ず最初に
このファイルを読んでから作業を始めてください。

本ファイルは**注 chu の取込作業と現代語訳整備**に特化した運用ルールです。
書き下し本文の取込は性霊集補闕鈔 全 112 entries（docx 基準）で完了済で、
今後変更予定はありません。書き下し取込期の膨大な運用履歴・OCR ノウハウは
`_archive/memos/CLAUDE_20260424_kakikudashi_complete.md` に保全済です。

## 基本方針

### 共有知識バンクであること
このリポジトリは、kaimyo-app（戒名・諷誦文・法話生成アプリ）だけのための
データベースではありません。将来、複数の仏教関連アプリで共有して使うことを
想定しています：

- 戒名・諷誦文・葬儀法話の生成アプリ
- 葬儀以外の日常の法話生成アプリ
- 仏教関連のブログ制作アプリ
- 真言宗教義の勉強会・学習用アプリ
- その他、これから作るかもしれない仏教関連アプリ

### データ蓄積の方針
- 特定のアプリ用に最適化したデータ抽出はしないこと
- できるだけ豊かに、できるだけ網羅的にデータを保存すること
- 葬儀には使えなくても、日常の法話やブログ、教義学習で価値がある内容も
  すべて取り込むこと
- 空海の言葉と真言密教の知識を、可能な限り完全な形で蓄積する

## 作業フロー（1 サイクル）

**1 篇の途中でも、貼られた分だけ毎回 push まで完結させる。**

1. ケンシン：Gemini OCR 出力をチャットに貼付（任意の範囲で OK）
2. Cowork：書き下し（JSON `ページ[].kakikudashi`）と突き合わせ
3. Cowork：OCR 誤認字を書き下しの漢字に置換（原則 12 適用）
4. Cowork：訂正済テキストを raw 末尾に追記（順序ずれ OK・並び替えしない）
5. Cowork：`commit_message.txt` を更新（今回の訂正内容を簡潔に日本語で）
6. Cowork：「`commit_push.bat` ダブルクリックをお願いします」とケンシンに依頼
7. ケンシン：`commit_push.bat` 実行 → commit + push
8. push 成功確認 → 次の貼付待ち

### 重要
- 毎サイクル push まで完結させる（commit を溜めない）
- JSON の `chu` フィールドには当面触らない（raw 蓄積が優先）
- 作業単位はケンシンが貼付する任意の範囲（篇の途中でも構わない）

## 自動処理モード（2026-04-24 確立・最優先）

ケンシン要望「単純に漢字の入れ替えが目的」を受けて、`_dev_references/process_cycle.py`
を導入。**1 サイクル 1 コマンドで完結**する。判定理由分類（RAW-A/B/C/E/L/O）も
人手の grep 確認も不要。

### 標準フロー（1 サイクル）

```
1. ケンシン：チャットに Gemini OCR 出力を貼付
2. Cowork：貼付を /sessions/.../paste.txt に保存
3. Cowork：python3 _dev_references/process_cycle.py --paste /sessions/.../paste.txt
   （--dry オプションで書込なしの dry-run 可）
4. Cowork：サマリ確認（訂正件数・各篇内訳）して push 依頼
5. ケンシン：commit_push.bat ダブルクリック
6. 次の貼付待ち
```

### process_cycle.py の自動判定ロジック

| chu_check 出力 | 自動処理 |
|---|---|
| 完全一致／語幹一致 | raw のまま |
| VARIANTS 異体字一致 | 書き下し字に置換して raw |
| 1 字違い近接（漢字→漢字）| 書き下し字に置換して raw |
| 1 字違い近接（漢字→ひら/カナ/句読点/活用語尾）| raw のまま（候補棄却）|
| 一致なし | raw のまま |

判定理由分類（RAW-A/B/C/E/L/O）は記録しない。シンプル化を最優先。
省略記法・列挙書式・字義論じ等は「一致なし → raw のまま」で自動的に正しく処理される。

### 篇名解決（自動）

raw 篇頭の【篇名】と JSON `篇名` の差異を吸収するため：
- ひらがな・カタカナ・空白・記号を除外して漢字のみで比較
- VARIANTS テーブルで字形を正規化（灌↔潅、雞↔鷄 等）
- 双方向部分一致＋最長マッチで idx 解決

例：raw「大唐神都青龍寺故三朝国師灌頂の阿闍梨恵果和尚の碑」 → JSON idx=12
「大唐神都青龍寺故三朝の国師潅頂阿闍梨恵果和尚の碑」 自動解決成功。

### raw 篇頭メタの自動形式

```
【JSON 篇名（フル）】
（idx=N・訂正 N 件）

一　見出し　解説…
```

人手で書く長文メタは廃止。

### commit_message.txt の自動形式

```
性霊集 注 raw 追記：idx=N1, idx=N2（訂正 N 件）

idx=N1 篇名: 訂正 X / そのまま Y
  ・注番 旧→新 (raw字↔書き下し字)
idx=N2 篇名: 訂正 X / そのまま Y
```

人手で書く判定理由は廃止。

### Cowork の役割（最小化）

- 貼付保存（heredoc 1 回）
- process_cycle.py 実行（1 コマンド）
- サマリ確認＋push 依頼（1 文）

raw 末尾照合・NULL バイト検証・差分判定・commit_message 執筆は全部スクリプト任せ。
1 サイクル 1〜3 分で完結する想定。

### 例外処理

- 篇名解決失敗 → サマリに「WARN: 篇名「...」未解決・スキップ」と表示
  → ケンシンに報告して JSON 篇名と照合
- chu_check.py の見出し境界判定が崩れる項目（句点で見出し続く等）
  → そのまま raw に渡る（後で気付いたら手動訂正）

### 旧運用との関係

下記「効率モード運用ルール」（E-1〜E-6）は process_cycle.py 導入で
ほぼ自動化された。ルール自体は判定ロジックの根拠として残すが、
日常作業ではほぼ意識不要。

## 効率モード運用ルール（2026-04-24 確立・最重要）

詩篇は差分が 10 件超えるのが普通。1 篇 1 サイクル＋過剰な文脈確認＋長文
commit_message を続けると、書き下し取込期と同じく時間が無限にかかる。
以下のルールで 1 サイクル 5〜10 分で完結させる。

### ルール E-1：chu_check.py の判定候補をそのまま採用
- 「raw 維持」候補なら raw 維持
- 「書き下し準拠」候補なら FIX-V/D/P で訂正
- 文脈 grep は **「判定候補が明らかに不自然」な時だけ**
- 詩篇では大半が raw 維持。1 件ごとの grep 確認に時間をかけない
- 運用 4 ルール第 3 条「判定候補そのまま採用 OK・境界例で悩まない」徹底

### ルール E-2：複数篇まとめ処理
1 貼付に複数篇含まれる場合、必ず 1 サイクルで完結させる：

```python
# 1) 篇分割（python re.split で 【篇名】区切り）
# 2) for ループで chu_check.py を篇ごとに連続実行
# 3) 判定一括 → raw 追記 1 回 → commit_message.txt 更新 1 回 → push 依頼
```

篇ごとに別サイクルにしない（オーバーヘッドが致命的）。

### ルール E-3：raw 篇頭メタは最小限

**必須**：日付不要・`（idx=N・書き下し準拠 N 件・raw 維持 N 件）`
**任意**：raw 維持件数が多いとき、1 行 1 件で簡潔に判定コード併記
**省略可**：過去サイクルのような長文の傍証パターン詳細列挙

例（詩篇典型）：
```
【篇名 詩型】
（idx=N・書き下し準拠 0 件・raw 維持 3 件）

一　見出し　解説…
```

### ルール E-4：commit_message.txt は 10〜25 行

構成（最大 5 セクション）：
1. 1 行目：取込範囲＋件数サマリ
2. ■ 取込範囲（idx ごとに 1 行）
3. ■ 突合：一致 N / 差分 N / 字形訂正 N / raw 維持 N
4. ■ 字形訂正（あれば 1 行 1 件）
5. ■ raw 維持（1 行 1 件・判定コード併記）
6. ■ 備考（順序ずれ・VARIANTS 候補があれば）

過去サイクルのような長大なメタ説明は不要。

### ルール E-5：差分判定は機械的に（判定フローチャート）

| パターン | 判定コード | 対応 |
|---------|-----------|------|
| 見出し末尾「…」「…（以下〜）」 | RAW-E | raw 維持（省略記法）|
| 見出し「X・Y」形＋両字書き下しにあり | RAW-L | raw 維持（列挙書式）|
| 解説文「X は…の意」「X、〜也」 | RAW-A | raw 維持（字義論じ）|
| 解説文「諸本では」「勝又本」「誤字か」 | RAW-B | raw 維持（底本差）|
| 解説文の典故引用『〜』内に対象字 | RAW-C | raw 維持（典故引用）|
| VARIANTS 異体字＋解説文傍証なし | FIX-V | 書き下し準拠で訂正 |
| 踊り字（営営↔営々） | FIX-D | 書き下し準拠で訂正 |
| 助詞・句読点差（、↔て） | FIX-P | 書き下し準拠で訂正 |
| 上記いずれにも該当せず | 候補そのまま | chu_check の判定に従う |

境界例（idx=3 注七 解頤型：字句両方ありで配列差）は raw 維持で OK。
深く考えない。

### ルール E-6：1 サイクルの bash チェーン

理想形（1 つの bash 実行ですべて完結）：

```bash
# Step 1: 貼付保存＋篇分割
cat > /sessions/.../paste.txt << 'PEOF' ... PEOF
python3 -c "<分割スクリプト>"

# Step 2: chu_check 連続実行
for IDX in N1 N2 N3; do python3 _dev_references/chu_check.py --idx $IDX --input ...; done

# Step 3: raw 追記＋commit_message 更新（同一 python 内で連続）
python3 << 'PYEOF' ... PYEOF
```

ケンシンへの確認質問はしない（順序ずれ・複数篇は CLAUDE.md ルールで自動処理）。

## 守るべき原則

### 原則 1：既存 JSON 構造を変更しない
total 112 entries・ページ構造・`篇名`・書き下し本文は変更しない。
触っていいのは `chu` フィールドのみ（当面は raw 保存のみ）。

### 原則 9：目次と本文題のズレは両方保持
古い本ではよくある。Cowork の判断で書き換えない。必要に応じて口伝や
異同として記録。

### 原則 12：注の漢字は書き下し準拠（最重要）
raw の漢字が書き下しと異なる場合、**書き下しに合わせて raw を訂正する**。
この作業が本サイクルの中心。

異体字の例（自動訂正対象）：
燈→灯・灌→潅・毗→毘・鳧→鳬・廈→厦・島→嶋・藪→薮・辯→弁 など

### 原則 12 例外：解説文傍証ルール
書き下しと raw で字が違っても、**解説文が当該字を明確に論じていれば
raw 維持**。これは「書き下し側が OCR 誤読しているかもしれない」シグナル。

書き下し OCR 誤読疑いの代表シグナル：

- パターン A：解説文が「X は〜の意」「X、〜也」と字義を論じる
- パターン B：「勝又本では Y」「諸本では」など底本差を明記
- パターン C：典故引用文中にその字が使われる

例（巻第五実例）：
- raw【求獲】／書き下し「求撥」→ 解説文「獲は極か」で獲を論じる → raw 維持
- raw【岨径】／書き下し「㠊径」→ 解説文「岨は山が険しいこと」→ raw 維持

### 原則 13：現代語訳作業中の注追加 OK
注 chu は**辞書的拡張可能フィールド**として運用する。現代語訳作業中に
Cowork が発見した典拠・歴史的背景・字義解釈・関連語彙などは、弘大全集六に
記載がなくても該当篇の chu に追加できる。末尾に `\n\n【見出し語】解説文。`
で追記。

### 原則 4：AI が勝手に作り出した言葉は使わない
raw にある字句を忠実に反映する。意味を推測して raw の字を変えない。

### 原則 6：段階的に進める
篇ごと・サイクルごとに commit。一度に大量処理しない（コンテキスト溢れ
防止）。

### 原則 11：ラベルより内容を信頼
引き継ぎメモが「idx=N の続き」「特定篇の訂正」のようなラベル参照で
位置を指しているときは、メモに載っている特徴的な一節を `grep` して
内容と一致するか確認する。ズレていたら作業を始めずケンシンに報告。

## Gemini OCR 自動補正リスク

Gemini は時に raw の字を標準字へ自動補正する：

- 事例：「放昿」（書き下し）を「放光」（標準字）に補正
- 対策：raw 見出し語を**必ず書き下しと整合確認する**
- 整合しない場合は原則 12 で書き下し準拠に訂正、または解説文傍証ルールで
  raw 維持判定

## chu フィールドの保存形式（JSON 反映時）

raw 蓄積が完了してから JSON `chu` フィールドへ反映する際の形式：

- 型：文字列（配列化しない・JSON スキーマは変更しない）
- 形式：`【見出し語】解説文`
- 区切り：`\n\n`（空行挟む）
- 括弧：半角 `()`
- 見出し語は弘大全集六準拠、ただし書き下し準拠で字形訂正
- 読み仮名「（○○）」は見出し語から削除（解説文内は維持）
- 漢数字 + 半角スペースは削除

## git 運用

### 書き込み系 git は .bat 経由（必須）
Cowork サンドボックスから `.git/` 内のファイルを操作できません
（Operation not permitted）。そのため commit / push / reset / stash / checkout
などの書き込み系 git 操作は、必ず Windows 側で .bat ダブルクリックで実行
します。サンドボックスからの git 実行は **読み取り系のみ**（log, status,
diff, show）に留める。

### .bat 内の日本語は禁止・必ず外部ファイル経由
cmd.exe は既定で Shift-JIS で .bat を読むため、UTF-8 で書かれた .bat に
日本語を含めるとコマンドが誤解釈されて実行エラーになります。

**必ず以下の形にする**：
- .bat の中身は ASCII・英語のみ
- 日本語のコミットメッセージは `commit_message.txt`（UTF-8）に書く
- .bat からは `git commit -F commit_message.txt` で読ませる

### JSON/raw 書き込み後の NULL バイト検証
サンドボックスから JSON や raw を上書きする際、元ファイルが新ファイルより
大きいと truncate が効かず末尾に NULL バイトが残ることがあります。
書き込み後は必ず：

```python
with open(path, 'rb') as f:
    raw = f.read()
print('trailing nulls:', len(raw) - len(raw.rstrip(b'\x00')))
# 0 でなければ破損
```

破損した場合は `git show HEAD:<path> > <path>` で復旧する。

### Windows マウント上のファイル削除
`rm` コマンドは Operation not permitted で失敗する。削除が必要なときは
Windows 側で `del /f /q` を使う（.bat 経由）。

### 定番 .bat（リポジトリ直下）
- **`commit_push.bat`** … メイン運用（commit + push を 1 ダブルクリックで完結）
- `1_commit.bat` … commit のみ
- `2_push.bat` … push のみ
- `verify.bat` … git log + status 確認
- `fix_index.bat` … git index 破損回復

### commit_push.bat の固定 stage 対象（5 ファイル）
毎回書き換えず以下を固定列挙：

```
git add data/kukai/shoryoshu_miyasaka.json
git add _dev_references/shoryoshu_vol5_kindle_ruby.md
git add _dev_references/kindle_ocr_rules.md
git add _dev_references/shoryoshu_gendaigoyaku_raw.txt
git add CLAUDE.md
```

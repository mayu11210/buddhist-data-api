### 静音方針（2026-04-25 確立）

ケンシン指示「結果報告の内容は表示しなくていい・解釈で迷う時だけ教えて」を
受けて、以下を厳守：

- **Cowork 下訳の全文をチャットに書き出さない**

    通俗化判断 等）

  一文」だけで完結する。idx=0〜2 のような解説的な突合報告は**禁止**

### 取込順序の原則
**idx 番号順に進める**（idx=0 → 1 → 2 → 3 ...）。「散文系推奨」のような
ラベル誘導で順序を飛ばさない。前メモの推奨候補も「同じ idx 範囲内での
選び方」程度に留め、巻番号や idx を飛び越えない。

### 現代語訳の品質方針

- **共有知識バンク方針**：補注は豊かに（典故・歴史背景・字義解釈）
- **直訳寄り**：並列構文を保つ・対句を崩さない
- **詩篇は各句 1 行ずつ訳出**（散文化しない）
- **典故補注は括弧書き**で濃密に（郭璞・葉公・北邙・王子晋・伯夷・老聃・

- **密教教学用語は原語維持**（三密・心王・遮那・阿字・無縁慈悲 等）。既訳が
  通俗化していても Cowork 訳を維持
- **伝記矛盾／逐字誤読は必ず修正**（既訳の方が正しいケースを特定して採用）
- **韻律は現代語で再現困難ゆえ意味重視**

## git 運用

### 書き込み系 git は .bat 経由（必須）
Cowork サンドボックスから `.git/` 内のファイルを操作できません
（Operation not permitted）。そのため commit / push / reset / stash / checkout
などの書き込み系 git 操作は、必ず Windows 側で .bat ダブルクリックで実行
します。サンドボックスからの git 実行は **読み取り系のみ**（log, status,
diff, show）に留める。

### .bat 内の日本語は禁止・必ず外部ファイル経由
cmd.exe は既定で Shift-JIS で .bat を読むため、UTF-8 で書かれた .bat に
日本語を含めるとコマンドが誤解釈されて実行エラーになります。

**必ず以下の形にする**：
- .bat の中身は ASCII・英語のみ
- 日本語のコミットメッセージは `commit_message.txt`（UTF-8）に書く
- .bat からは `git commit -F commit_message.txt` で読ませる

### JSON/raw 書き込み後の NULL バイト検証
サンドボックスから JSON や raw を上書きする際、元ファイルが新ファイルより
大きいと truncate が効かず末尾に NULL バイトが残ることがあります。
書き込み後は必ず：

```python
with open(path, 'rb') as f:
    raw = f.read()
print('trailing nulls:', len(raw) - len(raw.rstrip(b'\x00')))
# 0 でなければ破損
```

破損した場合は `git show HEAD:<path> > <path>` で復旧する。

### Windows マウント上のファイル削除
`rm` コマンドは Operation not permitted で失敗する。削除が必要なときは
Windows 側で `del /f /q` を使う（.bat 経由）。

### 定番 .bat（リポジトリ直下）
- **`commit_push.bat`** … メイン運用（commit + push を 1 ダブルクリックで完結）
- `1_commit.bat` … commit のみ
- `2_push.bat` … push のみ
- `verify.bat` … git log + status 確認
- `fix_index.bat` … git index 破損回復

### commit_push.bat の固定 stage 対象（5 ファイル）
毎回書き換えず以下を固定列挙：

```
git add data/kukai/shoryoshu_miyasaka.json
git add _dev_references/shoryoshu_vol5_kindle_ruby.md
git add _dev_references/kindle_ocr_rules.md
git add _dev_references/shoryoshu_gendaigoyaku_raw.txt
git add CLAUDE.md
```

変更のないファイルに `git add` しても no-op なので害なし。

**背景**：2026-05-03 G2-A 着手の commit_push.bat 実行で **phantom deletion 事故**が
発生した。`git add -u`（トラック済ファイルの削除を一括 stage する命令）が、
Windows ⇔ サンドボックスのファイルシステム同期タイミングで一部既存ファイルを
「削除されたもの」と誤認し、`data/mikkyou/ri.json` `sa.json` `sanskrit.json` 等の
辞書ファイル 23 件 + `lib/search.ts` `package.json` 等の他ファイル 7 件、計 30 件

寸前だった。

**対策（2026-05-04 commit_push.bat 修正）**：

1. **`git add -u` を削除**：トラック済ファイルの修正は既存の `git add <dir>/`
   行（data/kukai/・data/mikkyou/・_dev_references/ 等）でカバーされる。`-u` の
   役割（削除の自動 stage）は archive 操作の少数ケースでのみ必要だが、それは
   別途 dedicated な one-shot .bat で `git rm` を明示する運用に変更。
2. **Step 4.5 SAFETY CHECK 新設**：`git status | findstr /C:"deleted:"` で
   stage された削除を検出。1 件でも見つかれば `##### DANGER #####` と表示して
   commit を中止する（phantom deletion safety guard）。
3. **意図的な削除の運用**：archive 移動など本物の削除が必要な場合は専用の
   one-shot .bat を作成し、明示的に `git rm <path>` を書いて意図性を担保する。

**新フロー**：

```
Step 0   Clean stale locks
Step 1-2 Reset index to HEAD（既存）
Step 3   Stage target files（dir-level 中心・git add -u は廃止）
Step 4   Show staged diff (--cached --stat)
Step 4.5 SAFETY CHECK: deleted: 検出 → 中止
Step 5   Check commit_message.txt
Step 6   Commit
Step 7   Push
Step 8   Verify
```

**運用上の注意**：
- Step 4.5 で「DANGER: deletions are staged. Aborting commit.」が出たら、
  Cowork に状況を報告すること。phantom deletion か意図的な削除かの切り分けが必要。
- 意図的な削除は `archive_<name>.bat` のような専用 .bat に分離する。
- ASCII のみで .bat を書く原則は厳守（cmd.exe Shift-JIS 解釈で日本語が壊れる）。


### `_archive/` フォルダ
一度きり使ったスクリプトや過去の引き継ぎメモは `_archive/` 以下に隔離：
- `_archive/scripts/` … 過去の .bat（リセット作業用など）
- `_archive/memos/` … 過去の引き継ぎメモ・書き下し期 CLAUDE.md

日常作業では触らない。緊急時の参考資料として保全。

## 倉庫設計原則と文体モチーフ索引（2026-05-04 確立）

### buddhist-data-api は完全な倉庫である

本リポジトリの根本的な役割は **「仏教知識の保管庫」** である。データを丁寧に保管し、
タグを付ける。それ以上のことはしない。**「このデータはアプリ X に使える／使えない」
のような用途判断はしない**（キュレーション禁止）。**使えるかどうかは使う側が判断
する**。倉庫は判断を肩代わりしない。

派生原則：

- **アプリ固有の構造は倉庫に持ち込まない**：諷誦文の「初段／二段／三段／四段／五段」
  は kaimyo-app 固有の出力フォーマットであり、倉庫側のタグ・フィールドにしてはいけない。
- **アプリ用途で命名しない**：`fusonmon_*`・`kaimyo_*` のようなアプリ名空間を新規追加
  しない。データの自然な姿で命名する。
- **タグ軸を網羅的に整える**：倉庫の責務は「網羅性」と「タグの正確さ」にある。

### 役割分担（kaimyo-app 諷誦文生成を例に）

```
[kaimyo-app 諷誦文生成チーム]
 5 つの段別注文担当者（初段／二段／橋／三段／四段／五段）が
 別々のタグ組合せで素材を倉庫に注文 → 受け取り担当者が下ごしらえ
 → 段別文書生成担当が生成 → 全体まとめ
        ↓
[buddhist-data-api 倉庫]
 性霊集／秘蔵宝鑰／弁顕密二教論／菩提心論／三教指帰… の本文を
 多軸タグ付きで保管。注文に応じて候補を返す。「段」「硬さ UI」等の
 利用側固有の構造は持たない。
```

「段」「初段／二段／…」の概念は **kaimyo-app 側に閉じる**。倉庫側は段を意識しない。

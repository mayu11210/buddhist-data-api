# 文体モチーフ索引 設計メモ（2026-05-04 初版）

## 目的

buddhist-data-api を「**仏教知識の保管庫**」として位置付けし、保管されている各データに**多軸ハッシュタグ**を付与することで、kaimyo-app（諷誦文・戒名・法話生成アプリ）等の利用側がタグ検索で必要な素材を取り出せるようにする。

本メモは 2026-05-04 のケンシン Cowork 設計議論を畳み込み、以後セッションでの実装フェーズの設計指針を確定するために作成する。

---

## 1. 倉庫設計原則の再確認（2026-05-04 議論で更新）

### 大原則：buddhist-data-api は完全な倉庫である

- **倉庫はデータを保管し、タグを付ける**。それ以上のことはしない。
- **「このデータはアプリ X に使える／使えない」のような用途判断はしない**（キュレーション禁止）。
- **使えるかどうかは使う側が判断する**。倉庫は判断を肩代わりしない。

### 派生原則

- **アプリ固有の構造は倉庫に持ち込まない**：諷誦文の「初段／二段／三段／四段／五段」は kaimyo-app 固有の出力フォーマットであり、倉庫側のタグ・フィールドにしてはいけない。
- **アプリ用途で命名しない**：`fusonmon_*`・`kaimyo_*` のようなアプリ名空間を新規追加しない。データの自然な姿で命名する。
- **タグ軸を網羅的に整える**：倉庫の責務は「網羅性」と「タグの正確さ」にある。

### 役割分担（kaimyo-app 諷誦文生成を例に）

```
[kaimyo-app 諷誦文生成チーム]
 ┌─初段 注文担当者 ─┐  ┌─初段 受け取り担当者 → 初段 文書生成担当
 ├─二段 注文担当者 ─┤  ├─二段 受け取り担当者 → 二段 文書生成担当
 ├─橋  注文担当者 ─┼─→├─橋  受け取り担当者 → 橋  文書生成担当 ─→ 全体まとめ
 ├─三段 注文担当者 ─┤  ├─三段 受け取り担当者 → 三段 文書生成担当
 ├─四段 注文担当者 ─┤  ├─四段 受け取り担当者 → 四段 文書生成担当
 └─五段 注文担当者 ─┘  └─五段 受け取り担当者 → 五段 文書生成担当
        ↓ 各々が別々のタグ組合せで素材注文
[buddhist-data-api 倉庫]
 ・性霊集／秘蔵宝鑰／弁顕密二教論／菩提心論／三教指帰…
   の本文（書き下し+現代語訳+漢文）を多軸タグ付きで保管
 ・注文（タグクエリ）に応じて候補リストを返す
 ・「段」「硬さ UI」「故人タイプ UI」のような利用側固有の構造は持たない
```

「段」「初段／二段／…」の概念は **kaimyo-app 側に閉じる**。倉庫側は段を一切意識しない。

## 2. タグ軸の体系（9 標準軸＋6 特殊軸＋例外）

### 標準 9 軸（あらゆる素材に付けうる）

| `season` | 季節（故人没月マッチに使う） | `season:春` `season:夏` `season:秋` `season:冬` `season:通年` |
| `故人` | 故人属性 | `故人:在家男性` `故人:在家女性` `故人:僧侶` `故人:弟子` `故人:師匠` `故人:天皇` `故人:親王` `故人:妻` `故人:母` `故人:父` `故人:子` `故人:一般` |

| `密教` | 密教教学用語の含有 | `密教:三密` `密教:阿字` `密教:遮那` `密教:即身成仏` `密教:曼荼羅` `密教:金剛界` `密教:胎蔵` `密教:本不生` `密教:五智` `密教:加持` `密教:真言` `密教:六大無碍` `密教:三十七尊` |

### 特殊 6 軸（人生集約句プール用）

「故人の人生のすべてを表すと感じられる一句」（諷誦文の三段⇄四段橋・法話の人生総括等で使う）の素材には、上記 9 軸に加えて以下を付与する：

| 軸 | 説明 | タグ値の例 |
|---|---|---|
| `主題` | 句が捉える核心的な主題 | `主題:無常` `主題:師弟` `主題:孝心` `主題:慈愛` `主題:求道` `主題:智慧` `主題:仁` `主題:静寂` `主題:精進` `主題:本来性` `主題:即身成仏` `主題:菩提心` `主題:不二` |
| `人生像` | どんな生き方の人にこの句が合うか | `人生像:求道者` `人生像:養育者` `人生像:学究` `人生像:職人` `人生像:守護者` `人生像:苦難を歩んだ人` `人生像:静寂を愛した人` `人生像:与える人` `人生像:遍歴の人` |
| `感情調` | 句の感情的響き | `感情調:慰撫` `感情調:激励` `感情調:諦観` `感情調:希望` `感情調:厳粛` `感情調:包容` |
| `一句性` | 単独で人生を集約しうるか | `一句性:核心`（単独で集約可）／`一句性:補助`（文脈が必要）／`一句性:導入` |
| `引用形式` | 利用側が自動生成する導入句のヒント | `引用形式:大師曰く` `引用形式:経曰く` `引用形式:典故` `引用形式:無形式` |
| `含意の射程` | 人生のどの局面を射程とするか | `含意:全人生` `含意:晩年` `含意:臨終` `含意:転機` |

### 例外軸：成句

「諸行無常」「色即是空」「父母所生身、速証大覚位」のように、漢文のまま広く認知されている定型句は、書き下しに変換せず漢文形のまま保管してよい。

| 軸 | 説明 | タグ値 |
|---|---|---|
| `成句` | 漢文成句の定型句性 | `成句:famous`（漢文のまま使える）／無タグ（標準書き下し扱い） |

## 3. テキストフィールドのルール

| フィールド | 必須? | 内容 | 諷誦文/法話で使うか |
|---|---|---|---|
| `text_kakikudashi` | **必須** | 書き下し形（標準）／漢文成句（`成句:famous` 時の例外） | 使う（諷誦文本体に流し込む） |
| `text_gendaigoyaku` | **必須** | neutral な現代語訳（事例適応の起点） | 使う（kaimyo-app 側で事例に見合う表現に再構築） |
| `text_genten` | オプション | 漢文原文（大正蔵等） | **使わない**（学術参照・SAT 校正用） |

### 書き下しルール（厳守）

- 倉庫の素材本文は **書き下し文を主**とする
- 漢文（genten / 大正蔵原典等）は学術参照用にフィールドとして保持してよいが、**諷誦文生成・法話生成では使わない**
- ただし「諸行無常」「色即是空」「父母所生身」のような有名な漢文成句は、書き下しに変換せずそのまま使ってよい（`成句:famous` で識別）

### 「事例に見合う現代語表現」の役割分担

- buddhist-data-api 側 = neutral な現代語訳を保管（事例非依存）
- kaimyo-app 側 = 故人プロファイルに応じて現代語表現を再構築（事例依存）
- 倉庫は「neutral 現代語訳」を返すのみ。事例適応は利用側の責任。

- アプリ用途依存度（fusonmon_*・kaimyo_* は禁止）

- 既存構造との整合（`data/kukai/`・`data/templates/`・`data/mikkyou/` 等）

**現時点の推奨**：案 B（`data/indices/motifs.json`）または案 A の組合せ。性霊集だけのモチーフ索引としては案 A、複数著作横断のモチーフ索引としては案 B。次セッション以降で実装する際にケンシン判断で確定。

## 7. kaimyo-app 側の責務（参考・倉庫側からは触らない）

倉庫設計原則に従い、以下は **kaimyo-app 側の責務**。本メモには概要だけ記載し、詳細は kaimyo-app 側の設計メモに譲る：

- 諷誦文の段構成（敬白・初段・二段・三段・橋・四段・五段・結語）の定義
- 各段の標準長・硬さレベル別長さ
- 各段の注文担当者がどのタグ組合せで倉庫にクエリするか
- 受け取り担当者の下ごしらえロジック（候補絞り込み・重複検出・故人プロファイルマッチング）
- 文書生成担当の段別生成プロンプト
- 全体まとめの整合性チェック（同じ典故が複数段で重複しないか 等）
- 引用形式タグ → 導入句自動生成（`引用形式:大師曰く` → 「弘法大師のたまわく、」 等）

## 8. 残課題

### 設計フェーズ
- ファイル名・配置の最終決定（案 A〜D）
- 「硬さ」軸の倉庫原則上の妥当性再評価（kaimyo-app UI 由来のため、データ自然属性に分解する案：`漢語密度`・`対句密度`・`典故密度` 等）
- 「場面」軸の最小化（場面は用途寄りのため、データ自然属性として明らかな場合のみ付与）

### データフェーズ
- bodaishinron.json は kakikudashi のみで gendaigoyaku 未反映（候補 A 半完了状態）→ 反映完了後に菩提心論からの素材抽出可
- 御遺告未取込 → 橋プール拡張時に取込検討（要設計セッション）
- 経典本体（華厳経・法華経・大日経・理趣経・大智度論等）の取込検討 → 別途取込セッション必要
- 漢籍引用（論語・荘子・書経・詩経・史記等）の橋プール採否

### 実装フェーズ
- 検索 API の設計（既存 `/api/houwa/citations` 系列拡張 or 新エンドポイント）
- 索引ファイルのフォーマット最終確定（JSON / JSONL / SQLite）
- パフォーマンス（性霊集 500〜1000 素材 + 他著作 200〜500 素材で 1000〜2000 素材規模）
